# Section 1
# Claude Code とは

---

## Claude Code とは

- Anthropic開発の **エージェント型AIコーディングアシスタント**
- ターミナルで動作し、コードベース全体を理解して自律実行
- **「ここを直して」ではなく「この機能を実装して」** とタスクレベルで指示
- ファイル編集・コマンド実行・外部ツール連携まで一貫して実行

**主な用途:**
- 複数ファイルにまたがるリファクタリング
- テストの自動生成・実行・修正
- Gitワークフロー（コミット、PR作成）
- MCP経由の外部ツール連携（Slack / GitHub / Notion）

---

## Cursor vs Claude Code — 設計思想の違い

**結論: 両方を併用するのが最も効率的**

| 観点 | Cursor | Claude Code |
|------|--------|-------------|
| 設計思想 | IDE-first | Agent-first |
| リアルタイムTab補完 | 非常に強い | VSCode拡張経由 |
| 大規模リファクタリング | 中程度 | 非常に強い |
| MCP連携 | 40ツール上限 | 上限なし |
| Hooks（自動化） | なし | 全14イベント |
| Agent Teams | 単一レベル | 双方向協調（実験的） |

> Cursor → ユーザーが操縦しAIがアシスト
> Claude Code → タスクを委譲しAIが自律実行

---

## 3つのモデルと使い分け

| モデル | ID | 特徴 | 最適な用途 |
|--------|-----|------|-----------|
| **Opus 4.6** | `claude-opus-4-6` | 最高性能 | アーキテクチャ判断、難解なデバッグ |
| **Sonnet 4.6** | `claude-sonnet-4-6` | バランス型 | 日常コーディング（推奨） |
| **Haiku 4.5** | `claude-haiku-4-5` | 最速・軽量 | 大量ファイル処理、コスト節約 |

**選択の目安:**
- 迷ったら **Sonnet 4.6** — 速度と品質のバランスが良い
- 難しいタスクは **Opus 4.6** — 複雑なアーキテクチャ判断に
- コスト重視なら **Haiku 4.5** — 定型作業・サブエージェント向け
- セッション中に `/model` コマンドで切り替え可能

## インストールと認証

**推奨方法（Native Installer — 自動更新あり）:**
```bash
curl -fsSL https://claude.ai/install.sh | bash
```

**その他:** `brew install --cask claude-code`（手動更新）

**要件:**
- macOS 13+ / Windows 10+ / Ubuntu 20.04+、4GB RAM
- Claude Pro / Max / Teams / Enterprise（無料プラン非対応）

**初回起動:**
```bash
cd your-project
claude          # ブラウザで認証フロー開始
```

## tmux + ctコマンドで強力なワークフロー

**ctコマンド:** tmux + Claude Code の統合ラッパー

| 機能 | 効果 |
|------|------|
| セッション永続化 | SSH切断しても作業が継続 |
| `--dangerously-skip-permissions` 自動付与 | サンドボックス併用必須 |
| Agent Teams並列表示 | Split Panesで各エージェントが独立ペイン |

**基本操作:**
```bash
ct                         # 基本起動
ct "タスクの説明"            # プロンプト付き起動
ct --list                  # セッション一覧
ct --attach                # 再接続
ct --kill                  # 全セッション終了
```

# Section 3
# CLAUDE.md — AIへの指示書

---

## CLAUDE.md — AIへの永続的な指示書

- プロジェクトのルール・規約をAIに伝える **設定ファイル**
- Cursorの `.cursorrules` に相当
- セッション開始時に **自動的に読み込まれる**
- Markdown形式で記述

**CLAUDE.mdの3層自動マージ:**

| レベル | パス | 用途 |
|--------|------|------|
| Level 1 | `~/.claude/CLAUDE.md` | グローバル（全PJ共通） |
| Level 2 | `~/dev/CLAUDE.md` | ワークスペース（PJ横断） |
| Level 3 | `project/CLAUDE.md` | プロジェクト固有 |

3つが自動的にマージされて読み込まれる

---

## CLAUDE.md のベストプラクティス

- **300行以内**、150〜200指示が限界
- **削除テスト**: 「削除したらClaudeが間違いを犯すか？」→ 犯さないなら削除
- `@import`で分割: `@docs/git-instructions.md`
- **簡潔に**: 「TypeScript strict mode必須、any禁止」
- 具体例を含める（ディレクトリ構造、コード例）
- **優先度を明示**: 「最重要ルール（必須）」「推奨ルール」

```markdown
❌ NG（冗長）
TypeScriptを使う場合、必ずtsconfigでstrictモードを有効にして...

✅ OK（簡潔）
- TypeScript strict mode必須
- `any`型禁止
```

---

## 実例: シェル演算子禁止ルール

**問題:** Allowリストのパターンマッチが `&&` や `|` を含むコマンドに効かない

**ルール:** 1コマンド = 1回のBashツール呼び出し

| NG（禁止） | OK（正しいやり方） |
|-----------|------------------|
| `mkdir -p /path && git init` | `mkdir -p /path` → `git init` を順次実行 |
| `cat file \| grep pattern` | Grepツールを使用 |
| `echo "text" > file.txt` | Writeツールを使用 |
| `cd /path && ls` | `ls /path`（cdせず絶対パスを使用） |

## CursorとClaude Codeの共存設定

**CLAUDE.md を Source of Truth として管理:**

```bash
# setup-coexistence.sh で一括設定
# 1. CLAUDE.md を作成
# 2. .cursorrules → CLAUDE.md へのsymlink
# 3. .cursor/rules → .claude/rules からの参照
./setup-coexistence.sh
```

**結果:**
- 両方のツールで同じプロジェクトルールを共有
- ルール更新は CLAUDE.md の1箇所だけ
- `.cursorrules` は自動的に同期
