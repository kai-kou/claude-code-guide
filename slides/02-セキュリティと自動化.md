# Section 4
# 権限設定とセキュリティ

---

## 4つの権限モード（defaultMode）

`~/.claude/settings.json` の `permissions.defaultMode` で設定

| モード | 動作 | 推奨対象 |
|--------|------|---------|
| **ask** | すべて確認 | セキュリティ最優先・初心者 |
| **auto** | Allowリスト一致なら自動実行 | 推奨（バランス型） |
| **delegate** | Claudeが自律判断 | 上級者 |
| **plan** | 計画全体を表示し一括承認 | 複雑タスク向け |

**推奨パス:** `ask` → Allowリストを育てながら → `auto` → 慣れたら `delegate`

---

## Allow/Deny リスト — パターンマッチ

```json
{
  "permissions": {
    "allow": [
      "Bash(git *)", "Bash(npm *)", "Bash(python3 *)",
      "WebSearch", "mcp__slack-fast-mcp__slack_post_message"
    ],
    "deny": [
      "Bash(rm -rf /)", "Bash(sudo *)"
    ]
  }
}
```

**ルール:**
- `*` で任意文字列にマッチ
- **deny は allow より優先** — 誤った危険操作のブロックに使う
- MCP: `mcp__server_name__tool_name` 形式
- **シェル演算子（&&, |, >）を含むコマンドはマッチ不可**

---

## サンドボックス — OSレベルの隔離

```json
{
  "sandbox": {
    "enabled": true,
    "autoAllowBashIfSandboxed": true
  }
}
```

| 機能 | 効果 |
|------|------|
| ファイル書き込み制限 | ホワイトリスト以外をブロック |
| ネットワーク制限 | 指定ホスト以外への通信をブロック |
| 書き込み許可 | カレントディレクトリ(.)と /tmp/claude |

**autoAllowBashIfSandboxed:** サンドボックス内ならBash自動許可

---

## --dangerously-skip-permissions の正しい使い方

**全ての権限チェックをバイパスするオプション**

**危険性:**
- `rm -rf`, 機密ファイル読み書き, `git push --force` が確認なし実行

**使用条件（すべて満たすこと）:**
1. サンドボックスが有効（`sandbox.enabled: true`）
2. `permissions.deny` で危険コマンドをブロック済み
3. tmux環境などバックグラウンド実行が必要な場合

**用途:** `ct`コマンドで自動付与（サンドボックス併用前提）

---

## 機密ファイル保護 — 二段構えの防衛

**第1防衛線:** `settings.json` の `permissions.deny`
```json
"deny": ["Edit(.env)", "Edit(.env.*)", "Write(credentials.json)",
         "Read(secrets.json)", "Bash(rm -rf /)", "Bash(sudo *)"]
```

**第2防衛線:** `protect-sensitive-files.sh` Hook
- PreToolUse(Edit|Write) で自動チェック
- `exit 2` でブロック、`exit 0` で通過

**ブロック対象:**
`.env*`, `credentials.json`, `*.pem`, `*.key`, `~/.ssh/*`, `~/.gnupg/*`

> deny リストは allow より常に優先。片方が失敗してももう一方が保護を維持

# Section 5
# Hooks — 自動化の要

---

## Hooks — イベント駆動の自動化基盤

Claude Codeのイベントに応じてスクリプトを自動実行

```json
{
  "hooks": {
    "Stop": {
      "type": "command",
      "command": "bash ~/.claude/hooks/slack-feedback.sh"
    },
    "PostToolUse": {
      "type": "command",
      "command": "bash ~/.claude/hooks/post-edit-lint.sh",
      "matcher": "Edit|Write"
    }
  }
}
```

- **command型:** bashスクリプト実行（高速・定型処理向き）
- **prompt型:** LLMに自然言語で判断を委ねる（実験的）
- **matcher:** ツール名のフィルタ（正規表現対応）

---

## Hookイベント一覧（主要5つ）

| イベント | タイミング | 主な用途 |
|---------|----------|---------|
| **PreToolUse** | ツール実行前 | 検証・ブロック・事前チェック |
| **PostToolUse** | ツール実行後 | フォーマット・Lint・後処理 |
| **Stop** | 停止時 | 完了通知・Slack投稿 |
| **Notification** | 通知発生時 | macOS通知・外部連携 |
| **SessionStart** | セッション開始時 | コンテキスト再注入 |

その他: UserPromptSubmit, SubagentStop, PreCompact, PermissionRequest 等（全14種）

**exit code:** `0`=正常, `1`=エラー（自動修正試行）, `2`=ブロック

---

## 実例: Slack連携（slack-feedback.sh）

**フロー:**
```
Stop → Slack投稿 → リアクション待機(1分→3分→5分) → 指示変換
```

**リアクション→指示マッピング:**

| 絵文字 | 意味 | 変換後の指示 |
|-------|------|------------|
| :+1: | LGTM | コミット&プッシュ |
| :eyes: | レビュー依頼 | コードレビュー開始 |
| :arrows_counterclockwise: | やり直し | タスクをやり直す |
| :x: | 破棄 | 変更を破棄 |
| :rocket: | テスト | テストを実行 |

**トークン管理:** macOS Keychain推奨（環境変数での平文保存は非推奨）

---

## 実例: Lint自動チェック（post-edit-lint.sh）

**PostToolUse(Edit|Write)** で発火

```bash
case "${FILE_PATH##*.}" in
  md)   # YAMLフロントマター閉じチェック
  sh)   # bash -n 構文チェック
  json) # jq empty 構文チェック
esac
```

**対応フォーマット:**
- **Markdown:** YAMLフロントマターの閉じ `---` チェック
- **Shell:** `bash -n` 構文チェック
- **JSON:** `jq empty` 構文チェック

`exit 1` → Claude Codeが自動修正を試みる

---

## 実例: Push前チェック（pre-push-check.sh）

**PreToolUse(Bash)** で `git push` コマンドを検出

**チェック項目:**
1. 未コミット変更の確認
2. プッシュ対象コミットの表示
3. 高リスクファイル（`.env`, `.secret`, `.key`）の変更検出

**動作:**
- 高リスクファイル検出 → `exit 2`（ブロック・ユーザー確認要求）
- 正常 → `exit 0`（プッシュ続行）

---

## 実例: コンテキスト再注入（compact-reinject.sh）

**/compact 後の情報ロスを防ぐ**

SessionStart(matcher: "compact") で再注入:

| 再注入する情報 | 取得方法 |
|-------------|---------|
| プロジェクト名 | `basename $(pwd)` |
| 現在ブランチ | `git branch --show-current` |
| 直近5コミット | `git log -5 --oneline` |
| スプリント情報 | `.sprint-logs/sprint-backlog.md` |
| 未コミット変更 | `git status --short` |

**CLAUDE.mdにも保持ルールを記述:**
```markdown
## コンパクション時の保持ルール
/compact後も必ず保持: 現在のタスクID、未解決エラー、ディレクトリ構造
```
