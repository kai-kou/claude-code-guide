## 1. Claude Code とは

### Claude Code の概要

![Claude Code とは](assets/slides/slide-03.png)

Claude Codeは、Anthropicが開発したエージェント型AIコーディングアシスタント。最大の特徴は「ターミナルで動作し、コードベース全体を理解して自律的にタスクを実行する」点にあります。

従来のAIコーディングツールが「この行を修正して」というピンポイントの指示を前提としていたのに対し、Claude Codeは「この機能を実装して」「このバグを修正して」とタスクレベルで指示を出せる。ファイルの探索、編集、コマンド実行、テスト、コミットまでを一貫してAIが自律的にこなしてくれます。

> 「ユーザーは一切編集しない、AIが実装してできましたと結果を出す。それでオッケーかどうか」 ── 甲斐

つまり、開発者の役割が「コードを書く人」から「タスクを指示し、結果をレビューする人」に変わる。これがClaude Codeの根底にある設計思想。この話を聞いた時点で、会場の空気が少し変わった気がしました。

### Cursor vs Claude Code

![Cursor vs Claude Code](assets/slides/slide-04.png)

ここから、多くのメンバーが使っているCursorとの比較にかなりの時間が割かれました。結論から言うと、両者は「IDE-first」か「Agent-first」かという設計思想が根本的に異なります。

- **Cursor**: IDEの中にAIを組み込むアプローチ。ユーザーがエディタ上で操縦し、AIがリアルタイム補完やインライン編集でアシストする
- **Claude Code**: AIにタスクを委譲するアプローチ。ターミナルからタスクレベルの指示を出し、AIがファイル横断で自律実行する

わかりやすく言えば、Cursorは「自分で運転してナビがアシストしてくれる車」、Claude Codeは「行き先を伝えるとそこまで連れて行ってくれるタクシー」です。

> 「CursorからClaude Codeに移行したいと言わせるぐらいの魅力を触れるかもしれないコンテンツ」 ── 甲斐

ただし万能というわけではなく、GeminiやCodexを活用したい場面ではCursorに軍配が上がることもあるとのこと。大規模リファクタリングやHooksでの自動化はClaude Code、Tab補完やインライン編集はCursor。ざっくり言えば「大きなタスクはClaude Code、細かい編集はCursor」が現時点の使い分けです。

### 3つのモデルと使い分け

![3つのモデルと使い分け](assets/slides/slide-05.png)

Claude Codeでは、用途に応じて3つのモデルをセッション中に `/model` コマンドで切り替えられます。

| モデル | ID | 特徴 | 最適な用途 |
|--------|-----|------|-----------|
| **Opus 4.6** | `claude-opus-4-6` | 最高性能 | アーキテクチャ判断、難解なデバッグ |
| **Sonnet 4.6** | `claude-sonnet-4-6` | 速度と知性のバランス | 日常コーディング推奨 |
| **Haiku 4.5** | `claude-haiku-4-5` | 最速・軽量 | 大量ファイル処理、コスト節約 |

> 「先日Sonnet 4.6出たんで基本こいつをベースに」 ── 甲斐

日常的なコーディングにはSonnet 4.6をベースに使い、複雑な設計判断ではOpus 4.6に切り替え、Lintやテスト実行のような軽い作業ではHaiku 4.5でコストを抑える。この三段階の使い分けが、利用制限の最適化にもつながります。

## 2. セットアップ

### インストールと認証

![インストールと認証](assets/slides/slide-06.png)

推奨はcurlによるNative Installerです。

```bash
curl -fsSL https://claude.ai/install.sh | bash
```

> 実行前に https://claude.ai/install.sh の内容を確認することを推奨します。

この方法でインストールすると、バックグラウンドで自動更新が走ります。Homebrewやnpmでもインストールできますが、自動更新の恩恵を受けられないため非推奨。

> 「Homebrewとかでも入るんですけど自動更新されへんとかっていうところがある」 ── 甲斐

Claude Codeは週単位で新機能がリリースされるため、自動更新は必須。アカウントはClaude Pro / Max / Teams / Enterpriseのいずれかが必要です（無料プランは非対応）。

### VSCode拡張のセットアップ

![VSCode拡張のセットアップ](assets/slides/slide-07.png)

Claude CodeにはVSCode拡張機能が用意されています。拡張機能マーケットプレイスで「Claude Code」を検索してインストールするだけです。

### セカンダリサイドバーでの活用

![VSCode セカンダリサイドバーでターミナル利用](assets/slides/slide-08.png)

甲斐が推奨していたのが、セカンダリサイドバー（右側）への配置です。

> 「セカンダリーサイドバーにターミナルを引っ越してやってそこに固定表示させる」 ── 甲斐

Claude Codeのアイコンを右クリックして「Move to Secondary Side Bar」を選択。これにより、以下のレイアウトが実現します。

```
┌──────────────┬─────────────────┬──────────────┐
│ ファイルツリー │  エディタペイン    │ Claude Code  │
│ (Primary)    │                 │ (Secondary)  │
└──────────────┴─────────────────┴──────────────┘
```

`Cmd+Opt+B` でセカンダリサイドバーの表示・非表示を瞬時に切り替えられます。

### tmux活用のすすめ

![tmux活用のすすめ](assets/slides/slide-09.png)

> 「tmuxっていうターミナルのセッション連続化できたり、ペイン分割できる」 ── 甲斐

tmuxの最大の武器はセッションの永続化。SSH接続が切れても、ターミナルを閉じても、tmuxのセッションはサーバー側で生き続けます。長時間のリファクタリングをClaude Codeに任せて離席し、戻ってきたら結果を確認する。この安心感はtmuxなしでは得られません。

ペイン分割も見逃せない。複数のClaude Codeを並列で走らせたり、Agent Teams（複数AIの協調機能）の各エージェントをSplit Panesで並べて表示できるのもtmux環境ならでは。

### ctコマンド -- 独自ラッパーの実装

![ctコマンド](assets/slides/slide-10.png)

> 「専用のコマンドみたいなのはこういうことやりたいから実装してくれて」 ── 甲斐

甲斐が自作した `ct` コマンドは、tmux + Claude Codeの起動を簡略化する独自ラッパー。`ct "タスクの説明"` と打つだけで、tmuxセッションの作成から起動まで一発で完了します。

**注意**: このコマンドは `--dangerously-skip-permissions` を付与するため、**サンドボックスの有効化が必須**です（後述のセクション4を参照）。

このコマンド自体をClaude Codeに実装してもらったという点がポイントです。「こういうことやりたい」と伝えたら、Claude Codeがシェルスクリプトを書いてくれた。エージェント型アシスタントの面目躍如です。

### /init でプロジェクト設定

![/init でプロジェクト設定](assets/slides/slide-11.png)

`/init` を実行すると、Claude Codeがプロジェクト構造を分析し、最適化された `CLAUDE.md` を自動生成してくれます。

覚えておきたい基本コマンド:

| コマンド | 説明 |
|---------|------|
| `/init` | CLAUDE.mdの自動生成 |
| `/model` | モデル切り替え |
| `/plan` | Plan Mode（実装前に計画確認） |
| `/compact` | コンテキスト圧縮 |
| `/clear` | 会話リセット |

特に `/clear` はタスク完了ごとに実行する最重要コマンドです。

## 3. CLAUDE.md -- AIへの永続的な指示書

### CLAUDE.mdとは

![CLAUDE.md とは](assets/slides/slide-12.png)

CLAUDE.mdは、プロジェクトのルールや規約をAIに伝えるための設定ファイル。セッション開始時に自動的に読み込まれます。

> 「Cursor Rulesみたいなもん」 ── 甲斐

Cursorの `.cursorrules` に相当し、プロジェクト固有の技術スタック、コーディング規約、ディレクトリ構造、テスト方針などをMarkdownで記述します。

### 3層構造の自動マージ

![CLAUDE.md の3層自動マージ](assets/slides/slide-13.png)

> 「ルールに関してもCursorと同じようにグローバルルール、ユーザーグローバルルール、プロジェクトルールとかそういう発想」 ── 甲斐

| レベル | ファイルパス | 用途 |
|--------|------------|------|
| Level 1 | `~/.claude/CLAUDE.md` | ユーザーグローバル（全PJ共通の個人設定） |
| Level 2 | 親ディレクトリの `CLAUDE.md` | ワークスペース共通（複数PJ横断ルール） |
| Level 3 | `./CLAUDE.md` | プロジェクト固有（技術スタック・規約） |

セッション開始時にこの3層が自動的にマージされて読み込まれます。

### 書き方のコツ

![CLAUDE.md のベストプラクティス](assets/slides/slide-14.png)

> 「あんま書きすぎたら言うこと聞かんようになる」 ── 甲斐

公式の推奨は300行以内、150〜200指示が限界です。効果的なCLAUDE.mdを書くためのポイント:

- **300行以内に収める**: 「削除したらClaudeが間違いを犯すか？」テストで不要ルールを削減
- **`@import`で分割**: `@docs/git-instructions.md` のように外部ファイルに切り出す
- **簡潔に書く**: 「TypeScript strict mode必須、any禁止」のように一文で伝える
- **具体例を含める**: ディレクトリ構造やコード例を添えると精度が上がる
- **優先度を明示**: 「最重要ルール（必須）」「推奨ルール」のように重み付け

### Cursorとの共存設定

![CursorとClaude Codeの共存設定](assets/slides/slide-15.png)

> 「Cursorと共存はできるので、シンボリックリンク使ったら」 ── 甲斐

CLAUDE.mdをSource of Truthとして管理し、`.cursorrules` はCLAUDE.mdへのシンボリックリンクにすることで、ルールの一元管理が可能です。

```bash
ln -sf CLAUDE.md .cursorrules
```

### AGENTS.md -- 複数AIツール共存の新標準

![AGENTS.md](assets/slides/slide-16.png)

`AGENTS.md` は複数のAIツール（Cursor、GitHub Copilot、OpenAI Codex等）が共通で参照できるオープン標準として策定が進められています。Claude Codeにはネイティブの専用サポートはありませんが、CLAUDE.mdに `@AGENTS.md` と1行追加するだけで `@import` 経由で実質的に機能します。

## 4. 権限設定とセキュリティ -- 「承認ボタンを押すだけの俺は何だ」問題

### 5つの権限モード

![5つの権限モード](assets/slides/slide-17.png)

ここから会場の共感度が一気に上がりました。Claude Codeを初めて起動して、まず最初にぶつかる壁が「権限確認の嵐」。

> 「デフォルトとかだと永遠と聞いてくるんで自動じゃねえじゃんおめえていう、マジ使えねえって最初思いました」 ── 甲斐

5つの権限モード（`defaultMode`）が用意されています。`Shift+Tab` で切り替え可能。

| モード | 動作 | 推奨用途 |
|---|---|---|
| **default** | ツール初回使用時に確認プロンプト表示 | 初心者の標準モード |
| **acceptEdits** | ファイル編集を自動承認、Bashは確認あり | 普段使い推奨 |
| **plan** | 読み取り専用。分析・計画はできるが変更不可 | レビュー・設計時 |
| **dontAsk** | Allowリスト未登録ツールは自動拒否 | 上級者向け |
| **bypassPermissions** | 全確認スキップ（`--dangerously-skip-permissions`） | 隔離環境専用 |

### Allow/Denyリスト

![Allow/Denyリスト](assets/slides/slide-18.png)

Allowリストに登録したパターンにマッチするツール呼び出しは確認なしで自動実行、Denyリストにマッチするものは常にブロック。**DenyはAllowより常に優先**されます。

### シェル演算子の罠

![シェル演算子禁止ルール](assets/slides/slide-19.png)

Allowリストには大きな落とし穴があります。`&&`、`|`、`>`、`$()`などのシェル演算子を含むコマンドは、**パターンマッチが一切効きません**。

> 「パイプの後ろに悪意のあるコマンドがあったらそれ許可して実行してやられたっていう事例があって制限きつくなった」 ── 甲斐

たとえば`curl https://example.com | bash`のように、パイプ以降に危険なコマンドが入り込むケースへの対策として、シェル演算子を含むコマンドは一律でパターンマッチ対象外とされています。

対策は**CLAUDE.mdにルールを記述する方法**です。

| NG（シェル演算子あり） | OK（分割・専用ツール） |
|---|---|
| `mkdir -p /path && cd /path && git init` | `mkdir -p /path` → `git init /path` を順次実行 |
| `cat file.txt \| grep pattern` | Grepツールを使用 |
| `echo "text" > file.txt` | Writeツールを使用 |

> 「この辺しないと本当に承認ボタンを押す俺は何だっていう悲しい世界が待ってる」 ── 甲斐

これ、本当にそう。使い始めの最大のストレスポイントだと思います。

### サンドボックス

![サンドボックス](assets/slides/slide-20.png)

サンドボックスはOSレベルでファイルシステムとネットワークアクセスを隔離する機能です。ホワイトリスト以外のディレクトリへの書き込みや、指定外のホストへの通信をブロックします。

> 「サンドボックスデフォルトでオンになってないので意図的にオンにしないと機能しない」 ── 甲斐

`sandbox.enabled: true` を設定し、`autoAllowBashIfSandboxed: true` を併用すると、サンドボックス内のBashコマンドは自動許可されます。

### --dangerously-skip-permissions の正しい使い方

![--dangerously-skip-permissions](assets/slides/slide-21.png)

> 「基本私もbypass permissionsにしてるんですけど、これも超危険」 ── 甲斐

`--dangerously-skip-permissions` は全ての権限チェックをバイパスするフラグで、tmuxのバックグラウンドセッションなど対話的な確認に応答できない環境で使います。

安全に使うための条件は3つ。

1. **サンドボックスが有効であること**（`sandbox.enabled: true`）
2. **`permissions.deny`で危険コマンドをブロック済みであること**
3. **tmux等のバックグラウンド実行環境で、対話的な確認に応答できない場合に限定すること**

「権限スキップ + サンドボックスなし」は絶対に避けるべき組み合わせ。サンドボックスなしで使うと、ファイル破壊・機密情報の漏洩・Gitの暴走といったリスクに無防備な状態になります。

### 機密ファイル保護

![機密ファイル保護](assets/slides/slide-22.png)

`.env`、`credentials.json`、SSH秘密鍵といったファイルをClaude Codeが意図せず読み書きするリスクに対して、**二段構え**で防御します。

- **第1防衛線**: `settings.json` の `permissions.deny` -- DenyリストはClaude Code本体が参照する最も堅牢な防御
- **第2防衛線**: `protect-sensitive-files.sh` Hook -- PreToolUseイベントでファイルパスを検査し、機密ファイルへのアクセスを`exit 2`でブロック

一方が破られてももう一方が守る、多層防御の考え方です。

## 5. Hooks -- イベント駆動で自動化する

### Hooksとは

![Hooksとは](assets/slides/slide-23.png)

ここからが中級者向けの領域。HooksはClaude Codeの実行ライフサイクル中に発生するイベントに応じてスクリプトを自動実行する仕組みです。

> 「編集時とかコミット時とかプッシュ時とかそういうところでスクリプトを挟んでセキュリティチェックする」 ── 甲斐

設定方法について、甲斐は興味深いアプローチを紹介していました。

> 「私1回もこのsettings.json開いたことなくて、全部Claude Codeに書かせてます」 ── 甲斐

settings.jsonの構造を覚える必要すらなく、「Stopイベントでslack通知するHookを設定して」とClaude Codeに指示すれば、適切な設定を自動生成してくれる。設定ファイルの書き方をAIに聞くのではなく、AIに直接書かせるという発想がClaude Codeらしい。

### イベント一覧

![イベント一覧](assets/slides/slide-24.png)

実用上よく使う主要なイベントは5つです。

| イベント | タイミング | 主な用途 |
|---|---|---|
| **PreToolUse** | ツール実行前 | 検証・ブロック・事前チェック |
| **PostToolUse** | ツール実行後 | Lint・フォーマット・後処理 |
| **Stop** | Claude Code停止時 | 完了通知・Slack投稿 |
| **Notification** | 通知発生時 | macOS通知・外部連携 |
| **SessionStart** | セッション開始時 | コンテキスト初期化・再注入 |

他にも`UserPromptSubmit`、`SubagentStop`、`PreCompact`など多数ありますが、困りごとが出てきたタイミングで追加していくのがおすすめです。

### Slack連携 -- リモート操作の起点

![Slack連携](assets/slides/slide-25.png)

`Stop`イベントが発火するとSlackに完了通知が自動投稿され、そこから3つの方法でリモート操作ができます。

- **方法1**: 絵文字リアクションで次の指示を自動送信（thumbsupでコミット&プッシュ等）
- **方法2**: Remote Controlでセッションに直接接続（スマホから操作）
- **方法3**: @Claude でSlackから直接コーディング委任

### Remote Control -- スマホからClaude Codeを操作

![Remote Control](assets/slides/slide-26.png)

> 「リモートコントロールっていう機能が出て、スマホでローカルの端末の指示出せる」 ── 甲斐

`claude remote-control`で起動し、QRコードでスマホを接続すると、ローカルPCのClaude Codeセッションをスマホから直接操作できます。Max プラン限定のResearch Preview機能です。

> 「それやるためにもう常時稼働になっちゃいました。家だと電源落とさない」 ── 甲斐

PCの前にいなくてもコーディングが進む環境を構築したことで、マシンの電源を落とす理由がなくなったということです。極端な例かもしれませんが、Hooksによる自動化がワークスタイルそのものを変え得ることを示す好例です。

### Lint自動チェック

![Lint自動チェック](assets/slides/slide-27.png)

`PostToolUse`イベントで`Edit`または`Write`が実行された直後にLintスクリプトを走らせます。ファイルの拡張子に応じて、Markdownならフロントマターの閉じ忘れ、Shellなら`bash -n`で構文チェック、JSONなら`jq empty`で検証。Hookが`exit 1`を返すとClaude Codeが自動的に修正を試みます。

### Push前チェック

![Push前チェック](assets/slides/slide-28.png)

`PreToolUse`イベントでBashコマンドを監視し、`git push`を検出した際に安全チェックを実行。未コミット変更の確認、プッシュ対象コミットの表示、`.env`や`.key`など高リスクファイルの変更検出を行います。

exit codeの使い分け:

| exit code | 意味 | Claude Codeの動作 |
|---|---|---|
| 0 | 正常 | そのまま実行 |
| 1 | エラー | コンテキストにエラー情報を追加 |
| 2 | ブロック | 実行を中断、ユーザー確認を要求 |

### コンテキスト再注入

![コンテキスト再注入](assets/slides/slide-29.png)

`/compact`実行後に`SessionStart`イベントが発火し、プロジェクト名、現在のブランチ、直近5コミット、スプリント情報、未コミット変更といった重要な文脈情報を自動で再注入します。コンテキスト管理と密接に関連する重要なHookです。

## 6. コンテキスト管理 -- 「バカになる」前にやるべきこと

個人的に、この勉強会で一番「なるほど」と思ったのがこのセクション。

### コンテキストウィンドウの仕組み

![コンテキストの仕組み](assets/slides/slide-30.png)

> 「コンテキスト溢れた状態で指示したらバカになるとかっていうのはあるある」 ── 甲斐

「バカになる」現象の正体は、コンテキストウィンドウの溢れ。Claude Codeが一度に覚えていられる情報量には上限があります。中身はInput Tokens（指示やファイル内容）、Output Tokens（Claudeの応答）、Cache（キャッシュ済み情報）の3つ。

キャッシュには「Cache Write」（新規保存、通常コスト）と「Cache Read」（保存済み情報の再利用、**90%割引**）がある。CLAUDE.mdやよく参照するファイルはキャッシュに載ることでトークン消費を大幅に節約できます。5分以内に再アクセスするとCache Readとして処理されるため、この「5分ルール」を意識するだけでコスト効率が変わります。

### ステータスライン

![ステータスライン](assets/slides/slide-31.png)

> 「ステータスラインでリアルタイムにチャット指示する下側に出せる仕組みがあります」 ── 甲斐

ステータスラインは4行のダッシュボード形式で、Claude Codeの入力欄の下に常時表示されます。

> 「今どのプロジェクトおるか、どのブランチか、どのモデル使ってるで、コンテキストの使用率」 ── 甲斐

- **Line 1**: プロジェクト | モデル | コンテキスト使用率 | コスト
- **Line 2**: コンテキスト詳細（現在/最大、キャッシュ書込/読取）
- **Line 3**: 利用制限（5時間/7日間/Sonnet/Opus使用率）
- **Line 4**: メモリ使用量（CLI | App | VSCode）

### /compact と色の読み方

![/compact と色の読み方](assets/slides/slide-32.png)

**コンテキスト使用率**の色で危険度を把握:

- **緑（0-60%）**: 安全。そのまま作業を続行
- **黄（60-80%）**: 注意。`/compact` を検討
- **赤（80-100%）**: 即座に `/compact` or `/clear` 推奨

> 「コンパクトめちゃくちゃ時間かかるんで、なるだけここに食らわないように」 ── 甲斐

コンテキストが溢れてから `/compact` を実行するのは最終手段。そもそも溢れさせない運用のほうが効率的です。

### ベストプラクティス

![ベストプラクティス](assets/slides/slide-33.png)

> 「タスクごとに新しいセッションでやんのがベストプラクティス」 ── 甲斐

コンテキスト管理のポイント:

1. **ステータスラインの色を見て残量を把握する** -- 緑なら安心、黄で警戒、赤で即対応
2. **タスク完了後は `/clear`、途中なら `/compact`** -- 迷ったらこのルールに従う
3. **重要な情報はCLAUDE.mdと外部ファイルに逃がしておく** -- コンテキストに依存しすぎない
4. **大きなファイルは部分読み**（offset, limitを指定）-- 1000行全体読みと50行読みでは消費トークンが20倍違う

## 7. 利用制限の把握と対策 -- 「週末叩けねえ」を回避するために

### 制限の種類

![利用制限の種類](assets/slides/slide-34.png)

ここは正直、痛い話題。Claude Codeは無制限に使えるわけではありません。

> 「5時間制限があって100になったらその後5時間のリミット時間が来るまで使えなくなります」 ── 甲斐

> 「1週間の中で100になっちゃったらリセットされるまで使えなくなる」 ── 甲斐

3つの制限レイヤー:

1. **5時間ローリングウィンドウ**: 直近5時間の利用量
2. **7日間制限**: 週単位のトークン消費量
3. **モデル別制限**: SonnetとOpusそれぞれに独立した7日間の利用枠

### Max 5x/20xの注意点

![Max 5x/20x](assets/slides/slide-35.png)

> 「Max 5xと20xが5倍20倍なんじゃねえのって思ってたんですけど、実はそうじゃないみたい」 ── 甲斐

5時間ウィンドウの制限はおおむね倍率通りですが、週間のSonnet枠は実質3.5〜6倍程度のばらつきがあり、Opusの独立キャップはMax 5xでも週15〜35時間程度と厳しめ。OpusのコンピュートコストはSonnetの約5倍です。

> 「Opusだけ使ってたらあっちゅう間に制限食っちゃう」 ── 甲斐

> 「今日も70-80%とか行ってるんで多分週末どうしようかな、週末叩けねえっていう」 ── 甲斐

### 確認方法

![確認方法](assets/slides/slide-36.png)

ステータスラインのLine 3で**利用制限の使用率**を常時監視できます。こちらはコンテキスト使用率とは別の指標で、閾値も異なります。緑(〜70%)なら安全、黄(70〜85%)で要注意、赤(85%〜)は危険ゾーン。

### 最適化のコツ

![利用制限の最適化](assets/slides/slide-37.png)

- **キャッシュヒット率向上**: セッション開始直後にCLAUDE.mdやよく使うファイルを読み込ませてキャッシュに載せる
- **5分以内の再アクセス**でCache Read（90%割引）
- **Haiku活用**: Lint、テスト、検索などの軽作業はHaikuで
- **専用ツール優先**: cat/grepではなくRead/Grepツールを使う

### 制限到達時の対応

![制限に近づいたら](assets/slides/slide-38.png)

| 使用率 | 対応 |
|--------|------|
| 50%超 | Haikuモデルに切り替え（`/model haiku`） |
| 60%超 | タスクを複数セッションに分割 |
| 80%超 | リセット待ち、緊急性の低いタスクは後回し |

## 8. MCP -- 外部ツール連携と「MCP死んだ説」

### MCPとは

![MCP](assets/slides/slide-39.png)

MCP（Model Context Protocol）は、Claude Codeと外部サービスを接続するためのオープンスタンダード。Slack投稿、GitHub Issue操作、Notion更新などを自然言語の指示だけで実行できるようにします。

ただし、ここで甲斐さんから衝撃的な一言が飛び出しました。

> 「今MCPは死んだんじゃねっていう流れが来てて、CLIの方がいいよっていう」 ── 甲斐

この発言に会場がざわついた。MCPの課題は、ツール定義やレスポンスがコンテキストウィンドウを大量に消費する点。Google WorkspaceのCLIツールが出てきたように、CLIベースのツール連携であればコンテキストへの影響は最小限で済む。

とはいえ「完全に死んだ」わけではなく、Slack連携やGitHub操作のように、CLIでは代替しにくいユースケースではまだまだ実用的。必要なサーバーだけを厳選して使うのがポイントです。

### 設定方法

![MCPサーバーの設定方法](assets/slides/slide-40.png)

`claude mcp add` コマンドで簡単に追加できます。

| タイプ | 説明 |
|--------|------|
| `http`（streamable-http） | クラウドサービス連携向け。推奨 |
| `stdio` | ローカルプロセスとの通信（Playwright等） |
| `sse` | Server-Sent Events。廃止予定のため非推奨 |

### おすすめMCPサーバー

![おすすめMCPサーバー6選](assets/slides/slide-41.png)

- **Slack**: メッセージ投稿・検索・リアクション操作
- **GitHub**: Issue/PR操作・コード検索・レビュー
- **Sentry**: エラー監視・パフォーマンスデータ取得
- **Notion**: ページ作成・データベース操作
- **Playwright**: ブラウザ自動操作・E2Eテスト
- **Context7**: ライブラリ最新ドキュメント参照（AI幻覚防止）

おすすめMCPサーバーは以下の6つ。実際に使ってみて、なくてはならないものだけを残すのが賢い運用です。

## 9. カスタムスキル & エージェント

### カスタムスキル

![カスタムスキル](assets/slides/slide-42.png)

ここからは上級者向け。Claude Codeを使い込んでいくと、毎回同じ手順を伝えていることに気づきます。PRレビュー、スプリント開始、デプロイ前チェック...こうした繰り返しをスキル化するのがカスタムスキルです。

> 「よくやってる指示とか自分の中でルールがあるんだったらそれをスキル化する」 ── 甲斐

`~/.claude/skills/`（個人用）または `.claude/skills/`（プロジェクト用）に配置する `SKILL.md` ファイルで定義。`/review-pr` や `/sprint-start` のような再利用可能なコマンドが使えるようになります。

### SKILL.mdの書き方

![SKILL.md の書き方](assets/slides/slide-43.png)

`` !`gh pr diff` `` と書けば、スキル呼び出し時にコマンドの実行結果がプロンプトに動的に注入されます。`$ARGUMENTS` で引数を受け取ることも可能。

> 「基本的には私自分で書いたことがないので、全部エージェントに書いてもらいましょう」 ── 甲斐

「こういうスキルが欲しい」とClaude Codeに伝えれば、SKILL.mdを生成してくれます。AIに書かせて試しながら育てるアプローチが実践的です。

### カスタムエージェント

![カスタムエージェント](assets/slides/slide-44.png)

ここで整理しておくと、スキルとエージェントは似ているようで役割が異なります。

| 種別 | 定義 | 配置場所 |
|------|------|---------|
| スキル | 再利用可能なプロンプトテンプレート | `~/.claude/skills/` or `.claude/skills/` |
| エージェント | 独自コンテキストを持つAIサブプロセス | `.claude/agents/名前.md` |

エージェントは独自のコンテキストウィンドウ、システムプロンプト、ツール権限を持つ独立したプロセスとして動く。迷ったら、まずスキルとして作ってみて、独立したコンテキストが必要だと感じたらエージェントに切り替えるのがおすすめ。

### Agent Teams -- 複数AIの協調

![Agent Teams](assets/slides/slide-45.png)

勉強会で最も盛り上がったのがAgent Teamsの話題。ここで会場の空気が一変しました。

> 「エージェントチームズっていうエージェントをチームとして起動できるものがあります」 ── 甲斐

> 「チームリーダーのメインセッションがこのタスクをこの3人のメンバーでやってくれって言ったらチームが出来上がって」 ── 甲斐

Team Leadが指揮官として複数のTeammateにタスクを割り振り、共有タスクリストとメッセージングで連携。tmux環境で実行すると各チームメイトがSplit Panesに独立表示され、並列で作業が進みます。

> 「トークンバカ食いするんで」 ── 甲斐

チームメイトの数に比例してコストが増え、セッション再開もできません。現時点では実験的機能ですが、フロントエンド・バックエンド・テストの同時開発など、ハマる場面では大きな威力を発揮します。

## 10. Cursorからの移行

### 移行の基本ステップ

![移行のポイント](assets/slides/slide-46.png)

> 「Cursorで開いてたプロジェクトClaude Codeで開いてCursorの設定見てClaude Codeに最適化してくれって言ったらよしなに設定はしてくれる」 ── 甲斐

移行の基本ステップは3つ:

1. `.cursorrules` を `CLAUDE.md` に変換（構造化して300行以内に）
2. `.cursor/rules` を `.claude/rules` にsymlinkで共有
3. `settings.json` で権限・Hooks・サンドボックスを設定

`/init` コマンドを実行して、既存の `.cursorrules` の内容をマージするよう指示すれば、Claude Codeが設定をよしなに整えてくれます。

### タイムリープ戦略

![タイムリープ戦略](assets/slides/slide-47.png)

> 「タイムリープ戦略はClaude Codeではresumeかrewindコマンドを使う」 ── 甲斐

- `/resume`: 過去セッションの会話を復元して再開
- `/rewind`（Esc+Esc）: コードと会話を特定時点に巻き戻し
- `--fork-session`: 元のセッションを保護しつつ別アプローチを試す

> 「Bash操作してるもの、git pushとかフォルダーmkdirとかコマンド操作してるもんは巻き戻りまへん」 ── 甲斐

Write/Edit経由のファイル変更だけがチェックポイント対象。Bash操作は巻き戻しの範囲外です。

### 画像の扱い

![画像の扱い方](assets/slides/slide-48.png)

> 「プロジェクト内に画像置いてこれ見てくれっていう方が安全」 ── 甲斐

Cursorでは画像をチャットに貼り付ける形でしたが、Claude Codeではファイルパスを直接指定する方式が基本です。`Cmd+V` でペーストした画像は `/tmp/claude/` に自動保存されますが、プロジェクト内にコピーしておくのが推奨です。
