毎回同じ指示を打ち込んでいることに気づいたとき、スキルを覚える時期です。

「PRをレビューして。観点はセキュリティ・バグ・テストカバレッジで」——毎日 Claude Code に同じことを言っていませんか。Claude Code には、こうした定型ワークフローを `/コマンド名` 一発で呼び出す **スキル**、専門タスクを独立した AI サブプロセスに任せる **エージェント**、そして複数の Claude インスタンスを協調させる **Agent Teams** という仕組みが用意されています。

この記事では「毎日 Claude Code を使っているがスキルやエージェントはまだ自分で作ったことがない」という方を対象に、実際の定義ファイルと運用の工夫を交えて説明します。

---

## スキル——定型ワークフローを `/コマンド名` に変える

### スキルとは何か

スキルは「再利用可能なプロンプト」です。`SKILL.md` というファイルに手順を書いておくと、`/コマンド名` で呼び出せます。Claude に自然言語で「コードレビューして」と伝える代わりに `/review` と打つだけで、毎回同じ品質基準でレビューが始まる——そのイメージです。

勉強会では「スキルは手順書みたいなイメージ」という表現が出ましたが、まさにそれです。人間でいえば SOP（標準作業手順書）に相当するものを、Claude が読める形式で書いておく感じです。

スキルは [Agent Skills](https://agentskills.io) というオープン標準に準拠しているため、他の AI コーディングツールでも将来的に流用できる可能性があります。

### 配置場所

| 配置場所 | パス | 適用範囲 |
|---------|------|---------|
| Personal | `~/.claude/skills/<名前>/SKILL.md` | 全プロジェクト共通 |
| Project | `.claude/skills/<名前>/SKILL.md` | 現プロジェクトのみ |

どこに置けばいいか迷ったら、まず Personal に置いて全プロジェクトで試し、プロジェクト固有の内容が増えてきたら Project 側に移すのがお勧めです。

### SKILL.md の書き方

ファイルの構造は YAML フロントマター + Markdown 本文です。最小限の例から始めましょう。

```markdown
---
name: review-pr
description: PRのコード変更をレビューする。「PRレビュー」「コードレビュー」と言われた場合にも使用。
user-invocable: true
---

現在のブランチのPRをレビューしてください。

以下の観点でチェック:
1. バグの可能性
2. セキュリティリスク（OWASP Top 10）
3. パフォーマンスへの影響
4. テストの充足度
```

`description` フィールドは見た目以上に重要です。ここに書いた内容を Claude が読んで「この状況ではこのスキルを使うべき」と判断するからです。「PRレビューと言われた場合にも使用」のような自然言語パターンを加えると、Claude が自動的にスキルを呼び出してくれます。

### 主要フィールドの整理

| フィールド | 説明 | デフォルト |
|-----------|------|----------|
| `name` | スキル名。省略時はディレクトリ名を使用 | ディレクトリ名 |
| `description` | Claude が自動呼び出しの判断に使う説明 | 本文の最初の段落 |
| `user-invocable` | `/` メニューから呼び出せるか | `true` |
| `disable-model-invocation` | Claude の自動呼び出しを禁止するか | `false` |
| `allowed-tools` | スキル実行中に許可するツール | 全ツール |
| `model` | 実行モデル（`sonnet`, `opus`, `haiku`） | 継承 |
| `context` | `fork` で独立コンテキストで実行 | なし |
| `agent` | `context: fork` 時に使うエージェント種別 | なし |

呼び出し制御の組み合わせは次のようになります。

| 設定 | ユーザーから呼び出し | Claude が自動呼び出し |
|------|-------------------|-------------------|
| デフォルト | `/スキル名` で可能 | description に基づいて可能 |
| `disable-model-invocation: true` | `/スキル名` で可能 | 不可 |
| `user-invocable: false` | 不可（メニューに表示されない） | 可能 |

### 動的コンテキスト注入と引数

`` !`コマンド` `` 記法を使うと、スキル呼び出し時に実際のコマンド結果がプロンプトに注入されます。Claude が実行するのではなく、スキル送信前の前処理として実行される点がポイントです。

```markdown
---
name: review
description: PRのコードレビューを実行する
user-invocable: true
allowed-tools: Read, Grep, Glob, Bash
---

!`gh pr diff` の変更を以下の観点でレビューしてください。

1. バグ・ロジックエラー
2. セキュリティリスク（OWASP Top 10）
3. テストカバレッジ
4. 命名・コードスタイルの一貫性

問題がなければ「LGTM」、問題があれば具体的な修正案を提示してください。
```

`!` で始まるコマンドは Claude が読むプロンプトに展開されるため、毎回 `gh pr diff` を自分でコピーして貼り付ける手間がなくなります。

引数を受け取りたい場合は `$ARGUMENTS` を使います。

```markdown
---
name: explain
description: 指定されたファイルを解説する
argument-hint: [ファイルパス]
---

$ARGUMENTS のコードを解説してください。
```

`/explain src/auth/login.ts` のように使えます。

### SKILL.md が長くなってきたら

プロンプトが肥大化してきたら、`references/` ディレクトリに補足情報を分離できます。

```
my-skill/
├── SKILL.md           # メイン指示（500行以内推奨）
├── reference.md       # 詳細なAPIドキュメント
└── examples.md        # 使用例
```

SKILL.md 本文からリンクしておくと、Claude が必要なときだけ読み込みます。スキルの説明文が多くなると、コンテキストウィンドウの予算（全体の 2%、最大 16,000 文字）を圧迫するため、不要なスキルは `disable-model-invocation: true` にするか削除しておきましょう。

### スキル作成のハードルはゼロ

実際には「自分でスキルを書いたことがない。全部エージェントに書いてもらう」という使い方をしている人も多いです。Claude に「PRレビューを自動化するスキルの SKILL.md を作って」と頼めば、たたき台を一発で出してくれます。最初の一歩として、まずこれを試してみてください。

## エージェント——専門家に役割を与えて独立して動かす

### エージェントとは何か

エージェントは「専門家としての役割定義」です。スキルがメインセッション内で実行されるのに対し、エージェントは常に独立したコンテキストウィンドウで動きます。

独自のシステムプロンプト、ツール権限、モデル設定を持ち、Task tool を通じてメインエージェントからデリゲート（委譲）される形で起動します。エージェントは自分の仕事が終わったら結果をメインに返して終了します。

### 配置場所と定義ファイルの書き方

```
.claude/agents/<名前>.md    # プロジェクト固有
~/.claude/agents/<名前>.md  # 全プロジェクト共通
```

定義ファイルも YAML フロントマター + Markdown 本文です。

```markdown
---
name: code-reviewer
description: コードレビューの専門家。コード変更後に積極的に使用。
tools: Read, Glob, Grep
model: sonnet
maxTurns: 15
---

あなたはシニアコードレビュアーです。
以下の観点でコードの品質を確認してください。

1. バグの可能性
2. セキュリティリスク
3. 命名規則の一貫性
4. パフォーマンスへの影響
```

### 主要フィールドの整理

| フィールド | 説明 |
|-----------|------|
| `name` | エージェント名（一意の識別子） |
| `description` | 役割の説明。Claude がデリゲートの判断に使用 |
| `tools` | 使用可能ツールのホワイトリスト（省略時は全ツール継承） |
| `disallowedTools` | 使用禁止ツールのブラックリスト |
| `model` | 使用モデル（`sonnet`, `opus`, `haiku`, `inherit`） |
| `maxTurns` | 最大実行ターン数 |
| `permissionMode` | 権限モード（`default`, `plan`, `acceptEdits` 等） |
| `memory` | 永続メモリスコープ（`user`, `project`, `local`） |
| `isolation` | `worktree` で一時的な git worktree で実行 |
| `skills` | 起動時に注入するスキル一覧 |
| `mcpServers` | 使用可能な MCP サーバー |

### 最小権限の原則を必ず守る

エージェント設計で一番大切なのは、必要なツールだけを渡すことです。

```yaml
# 調査専用エージェント——読み取りのみ
tools: Read, Glob, Grep, WebSearch, WebFetch

# コーディングエージェント——全ツール
tools: Read, Glob, Grep, Bash, Edit, Write

# レビューエージェント——読み取り + 限定 Bash
tools: Read, Glob, Grep, Bash
```

調査専用エージェントに書き込みツールを渡さないことで、「調べるだけのつもりがコードを書き換えてしまった」という事故を防げます。

### 実践例：調査専用エージェント

```markdown
---
name: researcher
description: 技術調査を担当する。WebSearchとドキュメント分析に特化。
tools: Read, Glob, Grep, WebSearch, WebFetch
model: sonnet
maxTurns: 15
---

あなたは技術調査の専門家です。
コードを書くことは絶対にしないでください。
調査結果は「要約 → 推奨事項 → リスク」の構造で報告してください。
```

### 組み込みサブエージェント

Claude Code には最初から使えるエージェントがあります。自前で作る前に、これらで足りないか確認しましょう。

| エージェント | モデル | ツール | 用途 |
|------------|--------|--------|------|
| Explore | Haiku（高速） | 読み取り専用 | ファイル探索・コードベース調査 |
| Plan | 継承 | 読み取り専用 | Plan Mode 時のリサーチ |
| general-purpose | 継承 | 全ツール | 複雑なマルチステップタスク |

## スキル vs エージェント——どちらを使うか

「スキルとエージェントの違いがよくわからない」という声をよく聞きます。判断基準はシンプルです。

**まずスキルで作る**のが基本方針です。`/コマンド名` で呼び出せれば十分なケース、メインセッションの文脈を引き継いでほしいケース、定型手順を自動化したいだけのケース——これらはすべてスキルが適しています。

スキルを使ってみて「独立したコンテキストウィンドウが欲しい」「別のモデル設定で動かしたい」「並列で動かしたい」と感じたときに、はじめてエージェントに切り替えればいいです。

```
迷ったら → まずスキル
         ↓ 独立CWが必要だと感じたら
         → エージェント
```

具体的に言うと、PRレビューの手順を自動化したいなら `/review` スキルで十分ですが、「セキュリティ担当・パフォーマンス担当・テスト担当が並列でレビューする」という運用にしたいならエージェントが適しています。

## Agent Teams——チームとして協調する（実験的機能）

### Agent Teams とは

Agent Teams は、複数の Claude Code インスタンスが協調してタスクを並列実行する機能です。1 人の Team Lead が複数の Teammate を指揮し、共有タスクリストとメッセージングで連携します。

勉強会での言葉を借りると「チームメイト同士がメッセージのやり取りで議論が始まる。1 人じゃない感がある」という体験がこの機能の本質です。サブエージェントが「仕事を終えたら結果だけ返す」のに対し、Agent Teams はチームメイト同士が直接会話しながら作業を進められます。

### 有効化方法

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

settings.json に上記を追加するだけです。

### アーキテクチャ

| コンポーネント | 役割 |
|-------------|------|
| Team Lead | チームを作成し、タスクを割り振り、進捗を管理 |
| Teammates | 独立したコンテキストで並列に作業を実行 |
| Task List | チーム共有のタスク管理（作成・割り当て・完了） |
| Mailbox | チームメイト間の直接メッセージング |

### サブエージェントとの使い分け

サブエージェントは「結果だけ返してくれればいい」タスクに向いています。コストも低めで、シンプルな委譲に適しています。Agent Teams はより協調的で、チームメイト同士が直接やり取りしながら進める必要がある場面で使います。

チームメイト数に比例してコストが増えるため、実運用では **3〜4 人チームが現実的な上限**です。「最大並列は 5 程度が現実的。コストと制限を考慮して 3〜4 人チームが妥当」という知見が実際の運用から得られています。

### 表示モード

| モード | 説明 | 要件 |
|--------|------|------|
| `in-process` | メインターミナル内で動作。Shift+Down で切り替え | 追加設定不要 |
| `split panes` | tmux/iTerm2 で各チームメイトが独立ペインに表示 | tmux or iTerm2 |

tmux セッション内で実行すると自動的に split panes モードになります。視覚的に誰が何をしているか確認できるため、実際の開発では tmux との組み合わせがお勧めです。

### 主なユースケース

並列コードレビューが最もわかりやすい例です。セキュリティ担当・パフォーマンス担当・テストカバレッジ担当が同時にレビューを実施し、それぞれの観点での指摘をまとめてもらう使い方です。

他にも独立したモジュールの並列実装、複数の仮説による並列デバッグ調査、フロントエンド・バックエンド・テストの同時開発といった場面で効果的です。

スプリント運用への応用も可能で、`sprint-master` が Team Lead として起動し、`sprint-coder`・`sprint-tester`・`sprint-documenter` などのチームメイトにタスクを割り振るという形にできます。「PO として承認するだけ、実作業はエージェントに任せる」というワークフローがこれで実現します。

### Agent Teams のエージェント定義例

```markdown
---
name: sprint-master
description: スプリント全体のオーケストレーション
tools: Read, Glob, Grep, Bash, Edit, Write, Task, TeamCreate, TeamDelete, SendMessage
model: sonnet
maxTurns: 30
memory: user
---

スクラムマスターとして、チームメイトにタスクを分配し、進捗を管理してください。
```

### 既知の制限

実験的機能のため、以下の制限を把握しておいてください。

- `/resume` での再開時に in-process のチームメイトは復元されない
- 1 セッションにつき 1 チームまで
- チームメイトはさらにチームを作れない（ネスト不可）
- チームメイトがタスク完了のマーキングを忘れることがある
- VS Code 統合ターミナル・Windows Terminal・Ghostty では split panes が非対応

## 導入ロードマップ

「どこから始めればいいかわからない」という方向けに、段階的な導入手順をまとめます。

**Step 1 — 繰り返しを見つける（1 日目）**
自分が毎日 Claude Code に言っている定型の指示を書き出します。「PRレビューして」「テスト書いて」「この関数を説明して」など、3 つくらい出てくるはずです。

**Step 2 — 最初のスキルを作る（2〜3 日目）**
最も頻度が高い作業をスキルにします。Claude に「この作業を自動化する SKILL.md を作って」と頼むのが一番早いです。`~/.claude/skills/` に置いて、`/コマンド名` で呼び出せることを確認します。

**Step 3 — 動的コンテキスト注入を試す（1 週目）**
`` !`gh pr diff` `` のような動的注入を加えて、毎回の手作業をゼロに近づけます。`allowed-tools` でツールを制限して、スキルが想定外の操作をしないよう整備します。

**Step 4 — エージェントに分離する（2 週目以降）**
「独立したコンテキストで動かしたい」タスクが出てきたら、エージェントに切り替えます。調査専用エージェントから始めるのが安全です。ツール制限を読み取り系のみに絞り、コードを書かせないエージェントは副作用が少なくて安心です。

**Step 5 — Agent Teams を試す（余裕が出たら）**
並列処理のメリットが出る場面（複数モジュールの同時開発、複数観点での同時レビュー）が出てきたら Agent Teams を試します。まずは 2〜3 人の小さなチームから始めてください。

---

スキルを作り始めると「なんでこれを今まで毎回手で書いていたんだろう」という感覚になります。定型作業をスキルに変えるだけで、Claude との対話の質が変わります。まず `/review` か `/explain` あたりから試してみてください。
